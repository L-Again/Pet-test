# 接口测试 - 接口风格与文档解析

## 1. 一句话解释

接口测试前要先看懂接口属于什么风格、接口文档怎么写、请求和响应字段分别代表什么，然后才能设计测试用例。

## 2. 解决什么问题

这篇笔记主要解决三个问题：

- 传统接口和 RESTful 接口有什么区别。
- 接口测试完整流程是什么。
- 接口文档应该重点解析哪些信息。

## 3. 核心概念

### 传统风格接口

传统风格接口通常通过 URL 名称直接表达操作意图，例如 `getUser`、`addUser`、`deleteUser`。

![[Pasted image 20260706162818.png]]

特点：

- 常用 GET/POST 完成所有增删改查操作。
- URL 里经常包含操作动词。
- 状态码可能统一返回 200。
- 开发灵活，但规范性较弱。

### RESTful 风格接口

RESTful 接口强调“URL 表示资源，HTTP 方法表示动作”。

![[Pasted image 20260706163206.png]]

常见对应关系：

- GET：查询资源。
- POST：新增资源。
- PUT：修改资源。
- DELETE：删除资源。

RESTful 更适合前后端分离和团队协作，因为 URL、请求方法、状态码都有更清晰的约定。

## 4. 两种接口风格对比

| 对比项 | 传统风格 | RESTful 风格 |
| --- | --- | --- |
| 请求方法 | 主要使用 GET/POST | 严格区分 GET/POST/PUT/DELETE |
| URL 设计 | URL 同时包含资源和操作 | URL 只定位资源 |
| 操作表达 | 从 URL 名称看操作 | 从 HTTP 方法看操作 |
| 状态码 | 经常统一 200 | 更遵循 HTTP 状态码规范 |

判断方法：

- 如果 URL 中有 `getUser`、`addUser` 这类动词，通常偏传统风格。
- 如果 URL 是 `/users`、`/users/1`，并通过 GET/POST/PUT/DELETE 区分操作，通常偏 RESTful 风格。

## 5. 接口测试流程

![[Pasted image 20260706165604.png]]

接口测试一般按这个顺序推进：

1. 需求分析。
2. 接口文档解析。
3. 设计测试用例。
4. 准备测试脚本。
5. 执行测试并跟踪缺陷。
6. 生成测试报告。

可选扩展是持续集成，例如通过 Jenkins 定时执行接口测试并自动发送报告。

## 6. 接口文档解析

接口文档通常由后端开发编写，用来描述接口名称、功能、请求参数、返回结果、错误码等信息。

- 请求方法。
- URL。
- 请求头。
- 请求参数名称、类型、是否必填、约束条件。

## 7. 我的理解

接口测试不是拿到 URL 就直接发请求，而是要先看懂接口背后的规则：

- 这个接口用什么方式表达操作。
- 请求数据有哪些约束。
- 响应结果应该长什么样。
- 异常情况下应该怎么返回。

接口文档解析得越清楚，后面的测试用例越不容易漏。

## 8. 可行动建议

- 看接口文档时，先用表格整理请求方法、URL、请求参数、响应字段。
- 区分 HTTP 状态码和业务错误码，不要混在一起理解。
- 先判断接口风格，再理解 URL 和请求方法的含义。
- 对 RESTful 接口，要重点检查请求方法和资源路径是否匹配。

# 接口测试
---
## 传统风格接口
![[Pasted image 20260706162818.png]]
请求特点：
- 使用GET/POST可完成所有操作（增删改查）
- 不严格遵循HTTP协议规范（如可用GET删除数据）

URL设计特点：
- 同一操作有多个URL实现（如查询用户有3种URL）
- URL包含操作动词（如getUser/addUser）
- 开发人员可自由命名URL路径

状态码特点：
- 统一使用200表示成功
- 不区分具体操作类型（增删改查都用200）

优势：
- 通过URL名称可直接理解操作意图
- 开发灵活，只需要掌握GET？POST两种方法

典型示例
- 查询用户：GET/POST /getUser?id=1
- 添加用户：POST /addUser
- 删除用户：GET/POST /deleteUser?id=1

## RESTful风格接口
![[Pasted image 20260706163206.png]]
基本概念：
- 由Roy Fielding博士提出的设计风格
- 不是强制标准，而是设计原则和约束条件
- 全称：Representational State Transfer

核心思想：
- 面向资源的设计（URL只定位资源）
- 通过HTTP方法表达操作意图
- 利用HTTP状态码返回操作结果

请求方式规范
- GET：查询资源
- POST：新增资源
- PUT：修改资源
- DELETE：删除资源

URL设计特点：
- 唯一标识资源（如/users表示所有用户）
- 不包含操作动词（如不用getUser）
- 通过路径参数标识特定资源（如/users/1）

状态码规范：
- 200：查询成功
- 201：创建/修改成功
- 204:删除成功

CRUD对应关系：
- Create（增） → POST
- Retrieve（查） → GET
- Update（改） → PUT
- Delete（删） → DELETE

优势：
- 统一规范，便于团队协作
- 充分利用HTTP协议特性
- 前后端分离更彻底

## 两种风格对比
请求方式：
- 传统：主要适用POST
- RESTful：严格区分HTTP方法

URL设计：
- 传统：混合资源和操作
- RESTful：仅定位资源

状态码：
- 传统：统一200
- RESTful：遵循HTTP规范

识别方法：
- 查看请求方法是否严格对应操作类型
- 检查URL是否只包含资源路径

## 传统风格接口与RESTful风格接口对比
### 传统风格特征
请求方法：统一采用GET/POST实现所有操作（增删改查）

URL特点：
- 与资源不是一一对应关系
- 一个操作可对应多个URL（如查询用户有3种URL形式）
- 可从URL直接看出操作类型（如getUser表示查询，addUser表示添加）

状态码：统一返回200状态码

典型示例：
- 查询用户：http://127.0.0.1:8080/myweb/user/getUser?id=1
- 添加用户：http://127.0.0.1:8080/myweb/user/addUser
- 删除用户：http://127.0.0.1:8080/myweb/user/deleteUser?id=1

### RESTful风格接口特征
请求方法：
- 与HTTP协议方法严格对应
- GET-查、POST-增、PUT-改、DELETE-删

URL特点：
- 与资源一一对应（如/users表示用户集合，/users/1表示ID为1的用户）
- 不能从URL直接看出操作类型
- 需结合请求方法识别操作

状态码：
- 使用较为全面
- 查询成功：200
- 添加/修改成功：201
- 删除成功：204

典型示例：
- 查询用户：GET
  `http://127.0.0.1:8080/myweb/users/1`
- 添加用户：POST
  `http://127.0.0.1:8080/myweb/users`
- 删除用户：DELETE
  `http://127.0.0.1:8080/myweb/users/1`

## 传统风格与RESTful风格小结
传统风格：
- 仅使用GET/POST方法
- URL不唯一且包含操作语义
- 状态码统一返回200

RESTful风格：
- 使用全部HTTP方法（GET/POST/PUT/DELETE）
- URL唯一且仅定位资源
- 状态码使用灵活（200/201/204等）

核心区别：
- 传统风格：URL定义资源和操作混合
- RESTful风格：URL仅定义资源，不包含操作

# 接口测试流程
---
![[Pasted image 20260706165604.png]]
## 核心六步骤
- 需求分析：分析产品经理提供的需求文档
- 接口文档解析：理解开发编写的API文档（包含接口方法、参数、返回值等）
- 设计测试用例：编写Excel格式的测试用例
- 准备测试脚本：
	- 工具生成（如Postman）
	- 代码编写（Python+Requests+Unittest框架）
- 执行及缺陷跟踪：
	- 必须使用断言实现自动化判定
	- 发现并跟踪修复缺陷
- 生成测试报告：
	- 工具需安装插件
	- 代码需使用库（如HTMLTestReport）

可选步骤：持续集成（减少人工干预，实现全流程自动化）
- 使用Jenkins实现定时自动执行
- 自动发送测试报告邮件
- 大公司常用，中小企业可能不采用

## 注意
脚本≠代码：工具生成的脚本非编程语言编写

断言的作用：自动比对预期与实际结果

# 接口文档解析
---
## 解析目的
核心目标：明确需要从接口文档中提取的关键信息

文档作用：理解接口文档的定义及其在测试中的重要性

解析内容：确定需要从文档中提取的具体要素

## 解析流程
工作顺序：
- 完成接口文档解析
- 设计测试用例
- 准备测试脚本

## 接口文档
定义：接口文档（API文档）是由后端开发编写的，用于描述接口信息的文档。

编写者：通常由后端开发工程师负责编写和维护

核心功能：详细说明接口名称、功能实现、数据要求等关键信息

内容要素：包含请求参数格式、返回结果结构等完整接口规范

作用：
- 协同：实现团队人员工作协同配合
- 约束：确保项目更新修改时同步维护
- 描述：详细说明接口的功能、参数、返回值等信息

示例：
![[Pasted image 20260706171934.png]]

### 文档结构
项目说明：描述项目背景和架构

全剧错误码：
- 实际应为"状态码"，为避免与HTTP状态码混淆而称为错误码
- 成功时也可能返回（如0表示成功）
- 采用4位编码（如4001表示数据库查询错误）

修改记录：记录文档变更情况，确保团队同步

接口列表：
- 包含服务编号、名称、请求类型、URL和调用函数
- 示例：登录接口使用POST方法，URL为/api/v1.0/sessions

接口详情：
- 包含简要描述、请求参数、响应格式
- 示例：登录接口需要mobile和password参数
