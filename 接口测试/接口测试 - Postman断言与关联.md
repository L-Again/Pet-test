# 接口测试 - Postman断言与关联

## 1. 一句话解释

Postman 断言用来自动判断接口响应是否符合预期，Postman 关联用来把上一个接口返回的数据传给下一个接口使用。

## 2. 解决什么问题

手动看响应结果容易漏，也无法支撑批量执行。断言可以让 Postman 自动判断通过或失败。

接口之间经常存在依赖，例如登录后拿到 token，再用 token 请求其他接口。关联可以把这种依赖数据保存起来并复用。

Postman 断言是在 `Tests` 后置脚本中写 JavaScript 代码，在收到响应之后执行。

## 4. Postman 关联

接口关联用于解决接口之间的数据依赖。

典型场景：

- 登录接口返回 token，后续接口要使用这个 token。
- 添加员工接口返回员工 ID，查询或修改员工接口要使用这个 ID。

实现思路：

1. 先发送产生数据的接口。
2. 从响应数据中提取需要的字段。
3. 把字段保存到全局变量或环境变量。
4. 后续接口从变量中读取数据并发送请求。

## 5. 我的理解

断言解决的是“结果是否正确”的自动判断问题，关联解决的是“接口之间如何传递数据”的流程衔接问题。

如果只有请求，没有断言，接口测试仍然偏手工检查；如果不会关联，就很难测试真实业务流程。

## 6. 可行动建议

- 每个关键接口至少写状态码断言和核心字段断言。
- 对登录、创建数据这类接口，要练习提取 token、ID 等字段。
- 变量命名要清晰，例如 `token`、`employee_id`。
- 业务场景测试时，先画出接口依赖关系，再写关联。

## 断言工作原理
工作时机：断言代码的执行发生在收到HTTP响应之后，而不是在发送请求时立即执行。这是理解断言工作原理的关键点。

# Postman关联
---
定义：当接口和接口之间有依赖关系时，需要借助Postman关联技术来实现。

应用场景：
- 登录接口返回的令牌数据被添加员工接口依赖
- 添加员工接口返回的员工ID被查询员工接口依赖（作为路径参数）

技术特点：实现步骤固定，按套路操作即可完成关联

## 实现步骤
假定：接口B产生的数据，被接口A依赖
- 发送接口B请求，获取响应数据
- 将响应数据，放入公共容器（全局变量、环境变量）中
- 接口A从公共容器中，提取数据，发送请求
