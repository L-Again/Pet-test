# 功能测试 - 网络请求与抓包

## 1. 一句话解释

网络请求与抓包是功能测试中定位前后端问题、验证请求响应数据、检查接口行为和分析异常的重要基础能力。

## 2. 解决什么问题

功能测试不只是点页面。当页面异常、订单缺失、登录失败、请求参数错误时，测试人员需要知道：

- 浏览器访问服务器的过程。
- 请求和响应分别包含什么。
- HTTP 状态码大致代表什么。
- GET 和 POST 有什么区别。
- 为什么要抓包。
- 如何用抓包判断前端问题还是后端问题。

## 3. 网络访问过程

用户在浏览器输入 URL 后，基本流程如下：

1. 用户在浏览器输入 URL 并回车。
2. 浏览器将请求发送给服务器。
3. 服务器查找并处理请求数据。
4. 服务器返回响应数据。
5. 浏览器渲染并显示结果。

请求与响应关系：

- 请求：客户端向服务器发送消息。
- 响应：服务器向客户端返回数据。
- 先有请求，后有响应，二者构成完整网络交互。

## 4. 请求的组成

### 请求行

请求行包含：

- 请求方法
- URL
- 协议版本

格式可以理解为：

```text
请求方法 + URL + 协议/版本号
```

URL 结构包含：

- 协议
- 主机地址
- 端口号
- 路径
- 参数

### 请求头

请求头用于描述客户端属性，通常以 key-value 形式展示。

常见字段：

- `Accept-Encoding`：客户端支持的压缩类型，例如 gzip、deflate。
- `Host`：请求目标服务器地址。
- `Connection`：连接方式，例如 Keep-Alive。
- `Cookie`：客户端存储的会话信息。

### 请求体

请求体用于携带请求参数，例如登录时的账号密码。

基本格式：

```text
name=value
```

多个参数可以用 `&` 连接。

## 5. 响应的组成

### 响应行

响应行包含：

- 协议版本号
- 响应状态码
- 状态描述

示例：

```text
HTTP/1.1 200 OK
```

### 响应头

响应头描述服务器属性，以 key-value 形式展示。

常见信息：

- 服务器类型，例如 Nginx。
- 编程语言版本，例如 PHP/5.6.40。
- 字符编码，例如 UTF-8。

### 响应体

响应体存放服务器返回给客户端的实际数据。

常见格式：

- HTML 文件
- JSON 数据
- 图片文件
- 纯文本文件

Web 开发中最常见的是 HTML 和 JSON。

## 6. HTTP 状态码

### 2XX：成功

表示请求已成功被服务器接收、理解并处理。

常见状态码：

- 200 OK

### 3XX：重定向

表示需要客户端采取进一步操作才能完成请求。

典型场景：

- 地址变更
- 页面跳转

### 4XX：客户端错误

表示客户端请求可能发生错误，妨碍服务器处理。

常见示例：

- 404 Not Found：请求资源不存在。

错误责任通常在客户端，例如 URL 错误、未授权请求等。

### 5XX：服务器错误

表示服务器处理请求时发生错误或异常。

错误责任通常在服务器端，例如数据库连接失败、代码异常、服务器过载等。

## 7. URL 重点

原始截图参考：

![[Pasted image 20260706141221.png]]

完整格式：

```text
协议://主机地址:端口号/路径?参数名=参数值&参数名=参数值
```

字段说明：

- 协议：数据传输规则，HTTP 不加密，HTTPS 加密。
- 端口：应用程序标识，HTTP 默认 80，HTTPS 默认 443。
- 路径：资源在服务器上的位置。
- 参数：客户端传递的数据，多个参数用 `&` 连接。

## 8. HTTP 与 HTTPS

共同点：

- 都是网络传输协议。
- 都用于客户端和服务器通信。

区别：

- HTTP 是明文传输，不加密。
- HTTPS 使用 SSL/TLS 加密传输。
- HTTP 默认 80 端口。
- HTTPS 默认 443 端口。

应用场景：

- 公网环境必须优先使用 HTTPS 保障安全性。

## 9. GET 和 POST

### 相同点

- 都可以向服务器发送请求。
- 都可以携带参数。

### GET 特点

- 参数直接放在 URL 中，浏览器地址栏可见。
- 参数暴露在地址栏，相对不安全。
- 受浏览器地址栏长度限制，参数不能过多。
- 通常没有请求体。
- 适合查询类、无敏感参数的操作。

示例：

- 商品搜索。

### POST 特点

- 参数放在请求体中，需要抓包才能查看。
- 参数不直接暴露，相对安全。
- 请求体无明显长度限制，可传递更多参数。
- 通常用于提交表单、登录、注册、大数据量传输。

示例：

- 登录功能。

### 核心区别

- 可见性：GET 参数可见，POST 参数不直接可见。
- 安全性：POST 更适合处理敏感信息。
- 容量：POST 可传输更大数据量。

### 测试验证要点

- 通过抓包工具检查敏感功能是否使用 POST。
- 涉及用户隐私的功能必须使用 POST。
- 如果发现 GET 方法传输敏感信息，应及时反馈开发。

## 10. 抓包

### 抓包概念

客户端和服务器交互时，所有请求和响应都以数据包形式传递。

抓包就是通过工具拦截传输中的数据包，查看或分析请求行、请求头、请求体、响应行、响应头、响应体等信息。

### 为什么要抓包

- 防御性测试：防范恶意数据篡改。
- Bug 定位：判断问题是前端还是后端引起。
- 数据验证：检查请求和响应数据是否完整、正确。

## 11. 抓包工具使用场景

### 定位问题

用于区分前后端 Bug。

例如订单显示异常时，可以检查数据库写入与查询环节。

### 弱网测试

模拟网络延迟或丢包，验证功能稳定性。

### 绕过限制

突破前端验证，直接测试接口。

例如跳过手机号格式校验，直接提交请求。

## 12. 定位前后端 Bug

原始截图参考：

![[Pasted image 20260706151030.png]]

判断口诀：

- 无请求：前端问题，说明请求没有发出去。
- 有请求无响应：后端问题，说明服务端没有返回数据。
- 有请求有响应：继续检查响应内容，判断是前端显示问题还是后端数据处理问题。

还可以结合 HTTP 状态码进一步确定问题类型。

## 13. 学习重点

- 请求和响应是 Web 测试的基础。
- URL、请求方法、请求体、响应码都可能成为测试分析点。
- 敏感信息不应使用 GET 暴露在 URL 中。
- 抓包不是只为看接口，而是为了定位问题归属。
- 功能测试人员需要能判断问题在前端、后端、数据库还是网络链路。

## 14. 常见错误

- 只看页面结果，不看请求和响应。
- 遇到问题直接提 bug，不先判断前后端归属。
- 不理解 GET/POST 区别，无法发现敏感信息暴露问题。
- 看到 4XX、5XX 状态码不知道责任方向。
- 抓包后只截图，不分析关键字段。
