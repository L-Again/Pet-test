# 功能测试 - Web项目与测试环境

## 1. 一句话解释

Web 项目测试前，需要先理解项目运行环境、BS/CS 架构、前后台区别，以及开发、测试、生产环境为什么要分开。

## 2. 解决什么问题

这篇笔记解决的是功能测试前的项目环境认知问题：

- 什么是 Web 项目环境。
- Web 项目由哪些组件组成。
- BS 架构和 CS 架构怎么区分。
- 前台、后台、前端、后端分别是什么。
- 公司为什么需要开发环境、测试环境、生产环境。
- 测试人员进入项目后如何快速熟悉项目。

## 3. Web 项目环境

### 环境定义

环境是项目运行所需的软件和硬件组合。

一个网站能打开的前提是服务器环境已经部署完成并且处于运行状态。

### 环境组成

常见 Web 环境包含四大组件：

- 操作系统：例如 Linux、Windows，服务器常用 Linux。
- 数据库：例如 MySQL、Oracle 等关系型数据库。
- Web 应用程序：例如 Nginx、Apache，用来处理请求。
- 项目代码：包含前端和后端所有代码。

这些组件需要通过配置文件联动，保证 Web 应用程序能正确访问数据库和项目代码。

## 4. Web 项目

Web 项目的本质特征：通过浏览器访问。

例如淘宝网页版属于 Web 项目。

对比：

- Web 项目：通过浏览器访问。
- APP 项目：需要安装软件才能使用。
- PC 端和移动端只是安装平台不同，不影响项目类型判断。

## 5. 架构

### BS 架构

BS 架构由 Browser 和 Server 组成：

- Browser：浏览器。
- Server：服务器。

所有 Web 项目都属于 BS 架构。

### CS 架构

CS 架构由 Client 和 Server 组成：

- Client：客户端。
- Server：服务器。

需要安装客户端的 APP 项目通常属于 CS 架构。

## 6. 前台和后台

### 身份区别

- 前台：普通用户使用的系统入口。
- 后台：管理人员使用的系统入口。

### 开发区别

- 前端：HTML、CSS、JavaScript 等开发人员的工作范畴。
- 后端：Java、PHP 等服务器端开发语言的工作范畴。

- 数据库：专门存储项目数据。
- 项目代码：包含完整前后端代码逻辑。

## 8. 公司环境套数

### 核心问题

公司环境配置的核心，是解决开发、测试和用户使用之间的冲突。

典型场景：如果开发同时进行多个功能开发，且共用同一环境，就可能导致功能之间互相影响，测试工作无法正常开展。

### 开发与测试环境是否应该分开

必须分开。

原因：

- 开发持续提交新代码，会影响测试已提交功能的稳定性。
- 开发完成一个功能后，需要独立环境测试，同时不影响其他功能开发。
- 如果功能之间有关联，共用环境会导致一个功能异常影响另一个功能测试。

### 环境套数最低要求

原始截图参考：

![[Pasted image 20260703160020.png]]

最低配置：

- 生产环境：用户使用的稳定版本环境，必须存在。
- 测试环境：内部人员使用的测试环境，必须存在。

关键区别：用户环境必须保持稳定，不能因新功能开发而中断服务。

### 标准环境分类

- 开发环境：开发人员专用，用于功能开发。
- 测试环境：测试人员专用，用于功能验证。
- 生产环境：用户使用的线上环境。

## 9. 开发、测试冲突处理方式

### 时间隔离方案

- 开发周：开发人员独占环境进行新功能开发。
- 测试周：测试人员独占环境测试，开发仅允许提交 bug 修复。

### 空间隔离方案

- 开发、测试、生产环境完全独立。
- 允许开发和测试工作并行进行。

## 10. 熟悉项目

进入公司或项目组后，第一步是全面熟悉负责的项目，这是后续测试工作的基础。

### 已存在项目的信息来源

文档资源：

- 需求文档：最权威的项目功能说明。
- 使用说明书：已上线项目通常会有用户操作指南。
- 测试用例：老员工编写的用例会体现功能模块划分。

人员咨询优先级：

```text
测试老员工 > 产品经理 > 开发人员
```

注意：开发人员通常只了解局部功能逻辑。

### 新项目的信息来源

核心策略：紧盯产品经理，必须获取需求文档或明确需求说明。

如果文档缺失，应要求产品经理列出核心需求要点。

记忆口诀：

```text
新项目追产品，老项目问测试
```

## 11. 熟悉项目标准

测试人员熟悉项目至少要做到：

- 熟悉项目核心业务流程。
- 熟悉项目核心功能模块。

## 12. 功能分析思路

三步分析法：

1. 确定用户角色，例如普通用户、商家、管理员。
2. 分析用户使用场景和流程。
3. 识别缺失后会导致系统无法使用的关键功能。

## 13. 学习重点

- Web 项目测试前要理解环境和架构。
- 测试环境和生产环境必须区分。
- 熟悉项目时，先找文档，再问人。
- 老项目优先问测试老员工，新项目优先追产品经理。
- 功能分析要从用户角色、使用场景、核心流程入手。

## 14. 常见错误

- 不理解项目环境，直接开始测试。
- 把测试环境和生产环境混用。
- 只问开发，不看需求文档和历史测试用例。
- 熟悉项目只看页面，不分析用户角色和业务流程。
- 忽略后台管理系统，只关注前台用户入口。
