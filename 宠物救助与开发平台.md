# 一、项目概述与目标
---

项目名称：宠物领养与救助平台
项目性质：官方自营领养+社区互助救助公益平台
核心目标：解决流浪动物从发现（救助广场）到专业收容（领养中心）再到最终归宿（用户领养）的全链路闭环。以及前后彻底分离的动静资源管理架构


# 二、技术栈
---

前端：vue3 + Element Plus + Vite + Axios+  Vue Router
后端：Spring boot + Mybatis-Plus
数据库：Mysql 8.0
运行环境：Java 17  + node 25 + npm 11

# 三、项目架构
---

项目名称：宠物领养救助系统 
前后端分离架构 

前端：Vue

# 四、核心功能模块设计
---

## 账号与权限模块
### 1.角色视图隔离
- 普通用户：导航栏为`[logo] [首页] [救助广场] [领养中心]   ｜[头像下拉：个人中心（修改头像、昵称、个性签名、手机号；显示领养进度）、我的发布（管理自己在救助广场发布的宠物卡片）、退出登录]`
- 管理员：导航栏为`[logo] [首页] [救助广场] [领养中心]   ｜[头像下拉：个人中心（修改头像、个性签名、手机号）、管理中心（管理审核各项业务）、退出登录]`

### 2.登录注册
- 只有一个登录注册页面，通过数据库`users.role`字段确认登录角色信息，实时切换角色界面。
- 用户注册需填写手机号和用户名，后端给用户的role写入`role=1`。管理员账号由数据库预设。即：管理员`role=0`，用户`role=1`。
- 注册和登录模块统一在一个页面，通过模块的按钮"立即注册"和"立即登录"来进行模块视图切换。

## 救助广场模块
- 极简发布：用户只需要上传“现场图 + 位置 + 情况描述”，发布后，状态为：`status=0`（待审核）
- 管理员审核：管理员在管理中心可以看到用户发布的救助信息，点击“通过”则`status=1`（救助广场可见）；点击“驳回”则`status=2`（救助广场不可见，仅管理中心的”救助广场视图“和用户的“我的发布”页面可见）
- C2C互动：用户发布的宠物卡片，在点击宠物卡片后，宠物详情页侧重实时性，评论区允许用户发言
- 收编转化：若管理员决定官方介入，管理员可以在管理中心的救助广场视图下，将已经同意用户上传的救助信息的记录点击收编，修改状态为`status=3`(已收编)，救助广场的这个宠物卡片打上“已收编”的标签，用户无法再点击该卡片进入宠物详情页，`onClick`事件禁用，提示“该宠物已收编，无法查看”，管理员和用户都可以删除该记录，删除后，不再显示在救助广场，需注意，用户点击“删除”后，管理员的管理中心的记录不会删除，只有当管理员删除该记录后，才会从救助广场上下架。

## 领养中心模块
- 宠物档案卡片：由管理员手动发布，展示精修过的图片，健康报告，性格描述。
- 宠物详情页：类似电商详情页，左侧大图，右侧参数表，下方有描述信息，并且可以点击“申请领养”按钮。
- 状态控制：只有`status=0`（公示中）的宠物才可以展示，并可以显示“立即申请领养”按钮。
- 用户可以点击申请后，在自己的个人中心的领养状态中查看领养进度，申请成功与否都可以删除记录，但不影响管理员的管理后台中的保存的记录。

## 管理中心模块
### 对于救助广场视图：
- 展示待审核的用户在救助广场上传的帖子，可以选择“通过”或者“驳回”
- 可以对已经发布在救助广场的帖子选择“已收编”的操作，点击“已收编”后，自动在救助广场消失，后端查询救助广场列表时，过滤`status=3的数据`。

### 对于领养中心视图：
- 管理员可以上传发布新的宠物档案，其中包括，宠物图片，健康状况，文字描述。
- 管理员可以审核用户来自领养中心的申请，与用户进行交流，可以选择“同意”或者“驳回”

- 对上述两个模块中，保存所有记录，对于已经结束流程的记录，可以选择”删除“操作。
- 用户删除领养或上传记录不影响管理员中的记录，只有当管理员删除记录的时候，才是彻底删除。
- 对上述的两个模块管理分别采用了两个视图保存在一个管理中心的页面。

## 个人中心模块
### 用户个人中心：
- 展示用户的基础信息，并且可以设置修改头像、昵称、手机号、个性签名等操作。
- 展示用户的领养申请状况，其中采用步骤条的形式展示，具体参考状态机与数据库设计内容。

### 管理员个人中心：
- 展示管理员的基础信息，并且可以设置修改头像、昵称、手机号、个性签名等操作。

## 我的发布模块
- 用户可以查看自己发布的救助信息。
- 通过宠物卡片的形式展现。
- 对于管理员驳回的或者已收编的卡片信息用户可以选择“删除”操作。 

## 数据与存储架构
- 管理员上传的宠物档案的图片数据保存在`backend/uploads/pets/`目录里。
- 用户上传的救助广场的宠物信息图片数据保存在`backend/uploads/rescue/`目录里。
- 项目静态图片等全局素材保存在：`frontend/public`
- 项目页面组件等素材保存在：`frontend/src/views`

## 留言交互模块
- 用户与用户之间的交互采用评论区留言版的形式，显示时间轴，根据时间轴进行排序。
- 用户与管理员之间的交互采用对话框，根据发布时间交替显示。
- 对话框的大小固定，随着交互留言的内容增多，可以添加滚动条。

## 业务流程逻辑
该平台包含两条核心业务线：救助广场线与领养中心业务业务线。

### 救助广场业务逻辑：
- 普通用户上传流浪宠物信息 -> 管理员在管理中心的救助广场视图进行审核
	- 若审核”通过“展示在救助广场公示 -> 其他用户可评论联系 -> 管理员可以对其选择”收编“操作 -> 收编后的宠物卡片会被标记“已收编”的标记，并且不能再点击进入宠物详情页。
	- 若审核“驳回”则不显示在救助广场 -> 管理员和用户都可以选择“删除”该记录。

### 领养中心业务逻辑：
- 管理员在管理中心的领养中心视图上传发布新的宠物档案 - 用户提交领养申请 - 管理员与用户对话审核用户资格
	- 若管理员“同意”用户申请 -> 领养中心宠物卡片标记“已领养”的标签 -> 用户确认领养后宠物卡片从领养中心下架，流程闭环 -> 领养中心保留闭环的记录并且管理员可以选择“删除”操作。
	- 若管理员“驳回”用户申请 -> 宠物卡片继续显示在领养中心 -> 流程闭环 -> 用户可以在个人中心页面选择对这个记录进行“删除”操作。领养中心同样保留记录，并且管理员可以选择“删除”操作。

# 五、状态机与数据库设计
---
## 状态机逻辑闭环设计
### 救助广场状态机（rescue_request.status）：

| 状态  | 状态名称   | 触发事件               | 显示文案 | 后续状态         |
| --- | ------ | ------------------ | ---- | ------------ |
| 0   | 等待审核   | 用户上传救助信息（创建对象，状态0） | 等待审核 | 审核通过/驳回（1/2） |
| 1   | 审核通过   | 管理员点击同意（从状态0触发）    | 审核通过 | 救助广场展示（4）    |
| 2   | 审核驳回   | 管理员点击驳回（从状态0触发）    | 审核驳回 | 流程闭环         |
| 3   | 已收编    | 管理员点击收编（从状态4触发）    | 已收编  | 流程闭环         |
| 4   | 救助广场展示 | 审核通过自动触发（从状态1触发）   | 广场展示 | 管理员点击收编（3）   |
主流程：`0申请发布 → 1审核通过 → 4展示中 → 3已收编`
分支流程：`0申请发布 → 2驳回`

### 领养中心状态机（adoption_request.status）

| 状态  | 状态名称 | 触发事件               | 显示文案 | 后续状态         |
| --- | ---- | ------------------ | ---- | ------------ |
| 0   | 等待审核 | 用户点击申请领养（创建对象，状态0） | 等待审核 | 与用户沟通（3）     |
| 1   | 审核通过 | 管理员点击同意（从状态3触发）    | 审核通过 | 等待签收（4）      |
| 2   | 审核驳回 | 管理员点击驳回（从状态3触发）    | 审核驳回 | 流程闭环         |
| 3   | 沟通中  | 管理员点击开始沟通（从状态0触发）  | 沟通中  | 审核通过/驳回（1/2） |
| 4   | 运输中  | 系统自动触发（从状态1触发）     | 运输中  | 用户点击签收（5）    |
| 5   | 已签收  | 用户点击确认签收（从状态4触发）   | 确认签收 | 流程闭环         |
主流程：`0申请领养 → 3沟通中 → 1审核通过 → 4等待签收 → 5签收完结`
分支流程：`0申请领养 → 3沟通中 → 2审核驳回完结`

age VARCHAR(20) NOT NULL COMMENT '年龄', 
health_report TEXT NOT NULL COMMENT '健康情况', 
character_desc TEXT NOT NULL COMMENT '性格', 
images VARCHAR(500) NOT NULL COMMENT '宠物图片路径', 
status TINYINT NOT NULL DEFAULT 0 COMMENT '状态：0=公示中 1=已领养', is_deleted TINYINT NOT NULL DEFAULT 0 COMMENT '软删除：0=未删 1=用户删 2=管理员删',
